# v0.0.11 — PROGRAM/SETUP split, artifacts for both train+evaluate, startup spinner

> Three coordinated changes shipping as a single release. Each is internally
> small but they touch overlapping surfaces (system prompt, init flow, README),
> so we land them together.

## Problem

1. **`PROGRAM.md` mixes two audiences.** Today it holds both agent-steering
   sections (Goal, Target, Success, Notes) and tool-generation sections
   (Dataset, Evaluate, Contract). The whole file is injected into the agent's
   system prompt at run-start, diluting context the agent uses for ML
   research with content only `dojo task setup` needs.

2. **Artifacts don't reach the artifacts dir.** Per-experiment artifact dirs
   exist on disk (`.dojo/domains/{id}/runs/{eid}/artifacts/`) and the runner
   passes `artifacts_dir` to `evaluate()`, but `train()` never receives it.
   The agent doesn't know it exists. Result: agent invents broken code in
   `train.py` (e.g. reaching for `os.environ["DOJO_ARTIFACTS_DIR"]` that we
   never documented), evaluate's docstring says "optional — ignore" so
   AI-generated `evaluate.py` typically writes nothing, and the dir stays
   empty.

3. **Long silent pause at run startup.** Between `dojo run` and the first
   agent text the user sees nothing for ~tens of seconds. The CLI's event
   poller only fires after `orchestrator.start()` returns; meanwhile
   `start()` is loading the domain, indexing knowledge, building the system
   prompt, configuring the backend, and the Claude SDK is cold-starting its
   subprocess.

## Goals

- Agent's steering prompt is focused on the research goal, not data plumbing.
- Both `train()` and `evaluate()` can produce artifacts; they end up in the
  ArtifactStore and the active TrackingConnector exactly the way they already
  do for `evaluate`-only output.
- The "agent is starting" period is no longer a black box.
- Ship as one cohesive v0.0.11 release.

## Non-goals

- No automated migration of existing PROGRAM.md files. Single-tenant
  project; the README documents the manual split.
- No startup-speed optimization. UX-only fix; phase timings/profiling are
  out of scope.
- No frontend SSE phase events yet. Spinner is CLI-only; the frontend is
  currently de-prioritized per CLAUDE.md.
- No new artifact-related agent tools. Train/evaluate remain the only
  artifact-producing entrypoints (they're the only code that runs inside
  the runner subprocess and sees `DOJO_ARTIFACTS_DIR`).

---

## Architecture overview

```
.dojo/domains/{id}/
├── PROGRAM.md      ← agent steering (Goal / Target / Success / Notes)
├── SETUP.md        ← tool-gen spec  (Dataset / Evaluate)
└── runs/{eid}/
    ├── __dojo_train.py          (agent's train_code)
    ├── __dojo_runner.py         (framework-rendered runner)
    └── artifacts/               ← train() AND evaluate() write here
        ├── model.pkl            (e.g. agent's choice in train)
        └── residuals.png        (e.g. evaluate's diagnostic)
```

Post-run, `_ingest_artifacts` walks `artifacts/` and forwards every file to
`ArtifactStore.save()` and `TrackingConnector.log_artifact()` — that pipeline
is unchanged.

## Change 1 — Split `PROGRAM.md` into `PROGRAM.md` + `SETUP.md`

### File layout

| File | Audience | Sections | Read by |
|---|---|---|---|
| `PROGRAM.md` | Agent | `## Goal`, `## Target`, `## Success`, `## Notes` | `agents/prompts.py` (whole file → steering prompt) |
| `SETUP.md` | `dojo task setup` | `## Dataset`, `## Evaluate` | `tools/tool_generation.py` (whole file → generation prompt) |

Strict separation — tool generation does not see PROGRAM.md, the agent does
not see SETUP.md.

## Change 2 — `train()` and `evaluate()` both receive `artifacts_dir`

### Contract update

**`src/dojo/core/task.py`** — for `TaskType.REGRESSION`:

- Bump `contract_version: 3 → 4`. `assert_ready` rejects stale tasks with the
  existing re-verify message.
- Update `runner_callsite`:
  ```python
  runner_callsite=(
      'y_pred = train(X_train, y_train, X_test, '
      'artifacts_dir=Path(os.environ["DOJO_ARTIFACTS_DIR"]))\n'
      "    metrics = evaluate("
      "y_pred, "
      "X_train=X_train, X_test=X_test, "
      "y_train=y_train, y_test=y_test, "
      'artifacts_dir=Path(os.environ["DOJO_ARTIFACTS_DIR"]))'
  )
  ```
- Update `train_output_description` if needed (no signature change in the
  description string itself, but check — the registered description is
  shown in the prompt).
- Update `_REGRESSION_PROMPT` `## Train` section to reflect the new train
  signature so generated `evaluate.py` knows the same `artifacts_dir` is
  shared (currently no AI tool decision depends on this — but stating it
  avoids confusion).
- Update `_REGRESSION_PROMPT` `## Output` section: stop saying
  evaluate's `artifacts_dir` is "optional — ignore the parameter". Replace
  with: "If you have nothing better to save, default to a residuals
  scatter plot — three lines with matplotlib. Anything you write here is
  archived on every run, so make it consistent."

### Verifier update

**`src/dojo/core/task.py`** — `_REGRESSION_VERIFIER` already creates
`_artifacts_dir = _HERE / "artifacts"` and passes it to evaluate; no change
to the verifier (`train` isn't run by the verifier, only `load_data` and
`evaluate`).

### Agent prompt

**`src/dojo/agents/prompts.py`** — update the `_build_task_section` template:

- The "exact signatures" block changes train's signature:
  ```python
  # you write this:
  def train(X_train, y_train, X_test, *, artifacts_dir) -> y_pred: ...
  ```
- New `### Artifacts` mini-block under the contract:
  > Both `train()` and `evaluate()` receive `artifacts_dir: Path` — a
  > writable per-experiment directory. Anything written there is archived
  > and forwarded to the tracking backend (e.g. MLflow) automatically.
  >
  > **Train artifacts are opportunistic.** Use them when the artifact would
  > be worth comparing across experiments — a model checkpoint
  > (`joblib.dump(model, artifacts_dir / "model.pkl")`), a learning curve
  > plot, feature importances. Don't write to it on every run; only when
  > there's a real reason to keep the bytes.
  >
  > **Evaluate artifacts are durable.** They're produced for every run by
  > design and are the place to look for "what did this model actually
  > do" — residual plots, calibration, error breakdowns.
  >
  > Don't try to read prior experiments' artifacts from inside `train()` —
  > each run gets its own fresh `artifacts_dir`.

- Update the "Example train_code" block to show the parameter:
  ```python
  from sklearn.linear_model import LinearRegression


  def train(X_train, y_train, X_test, *, artifacts_dir):
      model = LinearRegression().fit(X_train, y_train)
      # optional: persist the trained model for later inspection
      # import joblib; joblib.dump(model, artifacts_dir / "model.pkl")
      return model.predict(X_test).tolist()
  ```

### `run_experiment` tool description

**`src/dojo/tools/experiments.py`** — update the `run_experiment` tool's
`description` and `train_code` parameter description so the JSON schema the
agent sees mentions `artifacts_dir`. Same wording as the prompt's `### Artifacts`
block, condensed.

### README

**`README.md`** — new `## Artifacts` subsection (or extend the Getting
Started block):

> Each experiment gets a fresh `.dojo/domains/{id}/runs/{eid}/artifacts/`
> directory. The runner passes its path as `artifacts_dir` to both `train()`
> and `evaluate()`.
>
> - **`evaluate(..., artifacts_dir)` writes durable per-run diagnostics**
>   (residual plots, calibration curves) — these are produced every run and
>   are part of the user-defined evaluation contract in `SETUP.md`.
> - **`train(..., artifacts_dir=...)` writes opportunistic artifacts**
>   (model checkpoints, training curves) when the agent decides they're
>   worth comparing across experiments. Not every run will write here.
>
> Everything written to `artifacts_dir` is:
> 1. Copied into the durable Dojo archive at `.dojo/artifacts/experiments/{eid}/...`
> 2. Forwarded to the active tracking backend
>    (`MlflowTracker.log_artifact` uploads to MLflow; `FileTracker` records
>    a reference; `NoopTracker` drops it).

### `dojo init` next-steps

Add one line to the printed next-steps block (after the `dojo task setup`
hint):

> `evaluate()` saves diagnostics to `artifacts_dir` every run; `train()` may
> save the model when worth it. See README §Artifacts.

## Change 3 — Phased startup spinner

### `orchestrator.start()` API

**`src/dojo/agents/orchestrator.py`** — add a `progress` parameter:

```python
async def start(
    self,
    prompt: str,
    *,
    domain_id: str,
    tool_hints: list[ToolHint] | None = None,
    require_ready_task: bool = True,
    progress: Callable[[str], None] | None = None,
) -> AgentRun:
    ...
```

Calls `progress(label)` (when not None) at four points, in order:

1. `"loading domain context"` — before `lab.domain_store.load(domain_id)`.
2. `"checking task readiness"` — before `TaskService(self.lab).assert_ready(...)`.
   (Combined with #1 if assert_ready proves consistently fast — keep as
   separate call for now; cheap.)
3. `"indexing prior knowledge"` — before
   `lab.knowledge_linker.get_domain_knowledge(domain_id)`.
4. `"configuring agent backend"` — before `self.backend.configure(...)`.

No call after backend.configure — control returns to the CLI, which
switches the spinner to its own "agent warming up…" label.

### CLI rendering

**`src/dojo/cli/run.py`** — wrap `orchestrator.start` and the start of
`orchestrator.execute` in a `Console.status` spinner:

```python
with console.status("[dim]starting agent run…[/dim]") as status:
    def _on_phase(label: str) -> None:
        status.update(f"[dim]{label}…[/dim]")
    try:
        run_obj = await orchestrator.start(
            prompt=prompt, domain_id=d.id, progress=_on_phase,
        )
    except TaskNotReadyError as e:
        ...
    status.update("[dim]agent warming up…[/dim]")
    set_current_run_id(base_dir, run_obj.id)
    # Existing welcome lines (run id, graceful-stop hint) stay outside the
    # status block — they're terminal output, not progress.

# Spinner exits before _stream_events takes over; first agent text replaces
# the warming-up line cleanly.
```

The `Console.status` context manager handles spinner draw/erase. When
exception is raised inside the block (e.g. `TaskNotReadyError`), the status
clears before the error message prints — ensures clean output on failure.

### Defaults / API surface

- `progress=None` is the default. The API path
  (`api/routers/agent.py` calling `orchestrator.start(...)`) and tests pass
  no callback and behave identically.
- Tests for `orchestrator.start` may add one assertion that progress is
  called when supplied (sanity check), but no test should require it.

### Out of scope (deferred)

Surfacing phases as `AgentEvent`s through the run record (so the SSE
stream and frontend get the same UX) is intentionally deferred. The
frontend is currently de-prioritized per CLAUDE.md, and the change is
trivial to add later — it would be an `event_type="phase"` event written
into `run.events` from each progress call, with the CLI subscribing to
those events instead of (or in addition to) the callback.

## Risks

- **Frozen v3 tasks won't re-run without re-verification.** This is by
  design (`assert_ready` rejects stale `contract_version`). Release notes
  call it out.
- **Manual PROGRAM.md migration.** Acceptable — single-tenant, low-blast
  radius, README documents it.
- **Spinner over async background task on Windows.** `Console.status`
  uses ANSI which is fine on macOS / Linux. If Windows ever lands as a
  target the spinner can degrade to plain prints — out of scope today.
